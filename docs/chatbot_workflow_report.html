Summary

What The Workflow Is Today

The chatbot is now built around a ContactThreadState that may contain multiple JobState records. Each inbound WhatsApp-style message is normalized, classified, assigned to a job, passed through structured extraction, optionally through quote, scheduling, or closure logic, then persisted back to storage together with an event trace.

The current flow is a hybrid. The orchestration and state transitions are deterministic Python logic, while some high-value interpretation layers can now be LLM-backed: intent classification, structured extraction, and reply drafting.

End-To-End

Runtime Message Path

1

Inbound message enters the workflow

The NiceGUI app or another caller creates an IncomingMessage and passes it into Agent.handle_incoming_message(...). This is the main entry point for the modern workflow.

2

The message is normalized

ingest_message() converts the inbound payload into a stored ConversationMessage with role user, timestamps, attachments, and source metadata. The message is immediately appended to thread history.

3

Intent and route are decided

The configured MessageClassifierPort determines an IntentType. The workflow converts that intent into a normalized route such as evidence_intake, quote_management, or scheduling.

4

A job is selected or created

The thread router either reuses the active job or opens a new one if the thread is dormant, terminal, or clearly represents a new inquiry. This is where the system keeps thread-level continuity but job-level separation.

5

Structured extraction updates the job

The configured extraction module reads the latest message and fills fields like name, address, city, installation type, dimensions, constraints, and other scope details.

6

Business modules run

Quote logic, mocked scheduling, mocked reminder creation, and closure updates are applied depending on the detected intent. If nothing else produces a reply, the workflow falls back to either asking for the next missing field or confirming that the case is ready for internal review.

7

Reply text is finalized and persisted

Outbound messages are appended to thread history as assistant messages, event records are created, thread and job status changes are recorded, and the full thread is saved to JSON storage.

Input Layer

How Messages Are Handled

Inbound structure

The workflow expects IncomingMessage objects with phone, text, timestamp, source, and optional attachments. This is the transport-facing shape.

Stored conversation shape

The inbound payload is transformed into ConversationMessage. That is the durable message shape used in history, the NiceGUI timeline, and LLM context assembly.

History behavior

Both user messages and assistant replies are appended into thread.history. This means the stored thread becomes the canonical source for chat replay and for recent-context prompts.

Source metadata

The workflow preserves message source labels such as whatsapp and workflow, which helps separate raw inbound input from system-generated replies.

Interpretation

How Intent Classification Works

Intent classification is delegated through MessageClassifierPort. The current system supports two implementations:

Mode Module How it works Current use
heuristic adapters/classifier/heuristic.py Keyword-based rules decide intents such as quote, scheduling, evidence, and new inquiry. Fallback or deterministic mode
llm adapters/classifier/llm.py The model-backed classifier evaluates the latest client turn using recent thread history. Preferred for stronger conversation understanding

After classification, the workflow maps the intent into a route: quote_management, scheduling, closure_followup, evidence_intake, or mixed. That route is stored as a first-class event.

Thread Logic

How Threads And Jobs Are Routed

A thread is the persistent contact record. A job is the work unit inside that thread. The router uses explicit rules from thread_router.py:

  • If no active job exists, create one.
  • If a new inquiry arrives while the active job is terminal, create a new job.
  • If a new inquiry arrives while there is already an active job, create a new job.
  • If the active job is older than 45 days, open a new job.
  • Otherwise keep using the active job.

This means the system currently favors opening a new job aggressively for fresh inquiries, while simpler follow-up messages remain attached to the active job.

Data Capture

How Data Is Extracted

Structured extraction updates the current JobState. The configured extractor works through StructuredExtractionPort.

Heuristic extraction

Regex and keyword rules parse names, addresses, cities, dimensions, urgency, constraints, and attachment evidence. This is deterministic and cheap, but shallow.

LLM extraction

The model receives the current job context and the latest client message, then returns a structured update with fields like contact_name, address, installation_type, dimensions, and constraints.

Missing field logic

After extraction, refresh_evidence_status() recomputes missing fields. The current required list is contact name, address, city, installation type, width, and height.

Status impact

If required data is still missing, the job becomes AWAITING_EVIDENCE. If all required data is present, the job moves to SCOPING.

Outbound Logic

How The Next Message Is Constructed

The workflow does not start by asking the LLM “what should I say?” Instead, it first resolves the business situation, then drafts the reply from that state. The current priority order is:

A

Disqualification reply

If the service is not an En Red Rosario case, the workflow sends the fixed NOT_QUALIFIED_MESSAGE.

B

Quote reply

If the intent is quote-related, quote logic may generate a rough or negotiated quote response. Right now the quote itself still comes from a mocked drafting adapter.

C

Scheduling or closure reply

Scheduling and closure modules return canned replies tied to mocked external actions.

D

Fallback workflow reply

If no other module produced an answer, the workflow either asks for the next missing field or says the case is ready for internal review. This is where the configured reply generator can polish the message.

Reply mode Module Behavior
rules adapters/reply_generation/rules.py Returns the fallback text unchanged.
llm adapters/reply_generation/llm.py Uses recent history plus current job state to rewrite the fallback into a more natural WhatsApp reply.

Important nuance: the LLM reply generator is currently applied only to the “no other reply was created” branch. Quote, scheduling, payment, and disqualification messages still come from deterministic business logic.

State Model

How State Evolves During The Conversation

Thread state

Thread status tracks high-level conversation posture: ACTIVE, WAITING_ON_CUSTOMER, WAITING_ON_INTERNAL, ESCALATED, or DORMANT.

Job state

Job status tracks the work unit itself: qualifying, awaiting evidence, scoping, quote sent, negotiating, scheduled, completed, payment pending, closed, disqualified, and more.

Typical intake progression

New job → qualifying → awaiting evidence → scoping → estimate ready. That is the most common path when the client is still sharing basic intake information.

Escalation triggers

The workflow escalates when there are multiple active jobs, multiple stakeholders, or active negotiation. Escalations push the thread into ESCALATED and are stored in both flags and events.

Observability

How Trace And Persistence Work

Every inbound turn writes both message history and workflow events into the JSON thread file. This is the basis for the NiceGUI technical trace panel.

Message history

Stored under thread.history. Includes user and assistant messages in chronological order.

Event stream

Stored under thread.events. Includes incoming_message, intent_detected, route_selected, extraction changes, outbound messages, status changes, quote events, scheduling events, and escalations.

Storage model

The default interactive setup writes one JSON file per thread under data/conversations/.

Persistence point

Persistence happens at the end of handle_incoming_message() through _persist(), after messages and event records have already been attached to the in-memory thread.

Reality Check

Important Current Gaps And Caveats

  • The workflow is modular, but not every module is equally intelligent yet. Quote, scheduling, and reminders are still placeholders.
  • Reply generation currently only polishes fallback evidence/scoping messages. Some other branches still answer with hardcoded text.
  • The service-request gate is still deterministic and keyword-driven.
  • The current extraction pipeline is message-centric, not full-thread synthesis. It updates from the latest turn, not from a global thread re-read.
  • The router uses simple heuristics for deciding when a new job should be created.
  • The system persists a rich trace, but there is no separate analytics or review database yet; everything lives in the thread JSON.
What this means operationally

The chatbot can now feel meaningfully more conversational, but the current state is best understood as a configurable workflow engine with selected LLM-powered interpretation layers, not yet as a fully autonomous sales and operations agent.
