Summary

The LLM Layer Is Now Real, But Partial

The chatbot’s conversation stack now includes real model calls in three places: intent classification, structured extraction, and reply drafting. These sit inside a deterministic workflow that still owns routing, state transitions, quote branching, scheduling branching, and persistence.

That means the model is not running the whole bot end to end. Instead, it acts as a set of bounded interpreters plugged into specific workflow steps.

Runtime Map

Where The Workflow Actually Calls Models

1

Intent classification

LLMMessageClassifier asks the model to map the latest client turn into one of the enumerated IntentType values. This happens early in the workflow, before route selection and job choice.

2

Structured extraction

LLMStructuredExtractionAdapter asks the model to return a typed update for the current job: contact name, address, city, installation type, dimensions, urgency, and constraints.

3

Reply drafting

LLMConversationReplyAdapter takes a deterministic fallback reply plus current job state and recent history, then rewrites that into a more natural WhatsApp answer.

The core orchestrator in workflow_agent.py still decides when each of these is invoked. None of the LLM adapters directly owns thread state or persistence.

Shared Transport

How The Base LLM Adapter Works

The low-level model transport is ClaudeAdapter. It implements the generic LLMPort and is currently configured directly against Anthropic models.

System prompt handling

The adapter receives the system prompt separately from the conversation messages and sends both to the Anthropic message API.

History handling

Prior user and assistant messages are sent as Anthropic message history. The system prompt remains separate from the turn history.

Stateless call pattern

Each call is handled directly through the Anthropic client. This keeps the adapter simple and avoids hidden cached state.

Event-loop safety

Calls now pass through run_coro_sync(...), which runs normally in synchronous contexts and transparently hops to a worker thread when an event loop is already running inside NiceGUI. 

run_coro_sync(coro):
  if no event loop is running:
    asyncio.run(coro)
  else:
    run asyncio.run(coro) inside a one-thread executor

Prompt 1

Intent Classifier Prompt

The classifier is constrained. It is not freeform chat generation. It returns a typed object with a single intent field. 

Clasificá el último mensaje del cliente dentro de estas intenciones:
new_inquiry, provide_evidence, quote_question, negotiation, scheduling,
rescheduling, payment, post_install, follow_up, chit_chat, unknown.

Pensá en conversaciones reales de En Red Rosario sobre redes de seguridad.
Si el cliente comparte un nombre, dirección, medidas, fotos o cualquier dato útil
del caso, eso cuenta como provide_evidence.

Context shape:

  • The current message text is passed as the main prompt input.
  • Up to the last 8 thread messages are added as message history, excluding the current message itself.
  • The classifier does not receive a separate large state summary yet; it relies mostly on recent dialogue.

Design implication: the classifier is better than keywords for ambiguous turns, but still fairly lightweight. It is not yet a full conversation analyst with deep thread memory.

Prompt 2

Structured Extraction Prompt

The extraction module asks the model for a typed job update, not prose. It returns a Pydantic object with fields like contact_name, address, city, installation_type, width_meters, and constraint arrays. 

Extraé datos estructurados del último mensaje del cliente para En Red Rosario.
Solo devolvé datos explícitos o muy confiables.
Si el mensaje parece ser solamente el nombre de la persona, cargalo en contact_name.
installation_type debe ser balcony, roof o stairwell cuando corresponda.
service_intent puede ser safety_net_installation si queda claro que busca redes de seguridad.

The actual model input also contains a manually assembled state summary, for example: 

Trabajo actual: nombre=..., direccion=..., ciudad=..., tipo=...,
ancho=..., alto=..., faltantes=...
Ultimo mensaje del cliente: ...

After the model returns, the workflow merges only the populated values into the current JobState. Existing values are preserved unless the new output explicitly updates them.

Important nuance

The extraction adapter currently reads only the latest message plus a compact summary of current job state. It is not doing full transcript re-synthesis. That keeps it fast, but it can miss broader conversational context.

Prompt 3

Reply Generation Prompt

The reply generator is the most openly conversational of the three model-backed modules. It still operates inside a controlled frame: the workflow provides the intended intent, route, state summary, known fields, missing fields, and a deterministic fallback reply. 

System prompt 1:
Sos el asistente virtual de En Red Rosario.
Respondé en español rioplatense, con tono humano, breve y claro para WhatsApp.
No inventes datos. Si faltan datos, pedí solo el próximo dato más importante.
No menciones procesos internos, rutas, estados ni IDs.
Tu objetivo comercial es avanzar la conversación sin abrumar.

System prompt 2:
Intent detectado: ...
Ruta seleccionada: ...
Trabajo actual: ...
Estado del trabajo: ...
Datos conocidos: ...
Campos faltantes: ...
Respuesta base sugerida: ...

Then the adapter appends recent thread history and finishes with an instruction like: 

Redactá la próxima respuesta al cliente.
Mantenela en 1 o 2 frases cortas.
Si la respuesta base ya es correcta, podés reformularla suavemente sin cambiar el objetivo.

This is a good example of the current design philosophy: the LLM is used as a controlled drafting layer, not as the sole owner of business intent.

Caveats

What Is Still Not LLM-Driven

  • Route mapping after intent classification is still deterministic.
  • Thread/job selection is still heuristic.
  • Service-request disqualification is still keyword-based.
  • Quote drafting is still mocked, so quote replies are not commercially intelligent yet.
  • Scheduling and reminders are still mocked operational placeholders.
  • Reply generation currently improves fallback intake/scoping messages, but not every branch of the workflow.

In short: the chatbot now has real conversational intelligence in important places, but the current system is still a hybrid workflow engine rather than a fully LLM-authored agent from start to finish.
