## Implementation patterns

The most conservative implementation is a **deterministic catalog-first state machine**. Here the bot uses a narrow intent taxonomy, resolves mentioned services against a catalog you own, and then runs a policy engine over structured fields. This maps cleanly onto the slot/parameter patterns exposed by current platforms: Lex slots are the values required to fulfill an intent, Dialogflow parameters use initial prompts plus reprompt event handlers during filling, Lex supports re-eliciting a slot by nulling it and looping back, and Rasa supports slot validation rules on collect steps. Current API guidance supports exactly this pattern: structured JSON outputs are explicitly supported, function calling lets the model interface with application tools via schemas, and current governance guidance emphasizes built-in guardrails, policy enforcement, and observability before execution. ## Recommended architecture for your module

The architecture I would recommend for this module is a **catalog resolver + policy engine + stateful quote flow**. The catalog resolver should map every mentioned service item to one of four statuses: `in_scope`, `adjacent_unsupported`, `unknown`, or `meta` such as help/cancel/human-agent. The policy engine should then decide whether to continue, clarify, reject only the extra, or hand off.
